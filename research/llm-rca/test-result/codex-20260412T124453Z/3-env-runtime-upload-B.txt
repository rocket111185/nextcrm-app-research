The incident was triggered by the change in [app/api/upload/presigned-url/route.ts](/home/user/nextcrm-app-research/app/api/upload/presigned-url/route.ts:12), introduced in commit `0dc39fc` (`Tighten upload URL handling`, Apr 11, 2026 22:26:52 +0300). That change replaced simple string concatenation with:

```ts
const publicBaseUrl = new URL(MINIO_PUBLIC_URL!);
```

At incident time, `MINIO_PUBLIC_URL` resolved to an empty string, because [lib/minio.ts](/home/user/nextcrm-app-research/lib/minio.ts:19) maps it from `process.env.NEXT_PUBLIC_MINIO_ENDPOINT`, and the repo’s `.env.local` currently has:

```env
NEXT_PUBLIC_MINIO_ENDPOINT=
```

`new URL("")` throws `TypeError: Invalid URL`, which matches the production stack trace exactly, including `input: ''`.

**Why This Broke Now**

Before the change, the route built the file URL with string interpolation:

```ts
`${MINIO_PUBLIC_URL}/${MINIO_BUCKET}/${key}`
```

That was unsafe but non-throwing when `MINIO_PUBLIC_URL` was blank. After the refactor, the same missing env var became a hard runtime failure, causing `POST /api/upload/presigned-url` to return `500`.

**Contributing Factors**

- `NEXT_PUBLIC_MINIO_ENDPOINT` is not validated at startup, while other MinIO vars are validated in [lib/minio.ts](/home/user/nextcrm-app-research/lib/minio.ts:3).
- The non-null assertion `MINIO_PUBLIC_URL!` suppresses TypeScript checks but does not protect against an empty string.
- The incident timestamp, `2026-04-11T23:16:55.311Z`, is about 3h50m after the refactor landed, which fits a deploy-regression timeline.

**Conclusion**

The most likely root cause is a missing or empty `NEXT_PUBLIC_MINIO_ENDPOINT` in the runtime environment, exposed by the new `buildPublicFileUrl()` logic added in the last change.
