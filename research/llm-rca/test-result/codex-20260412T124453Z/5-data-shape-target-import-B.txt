**Most Likely Root Cause**

The import regression was introduced by the new `parseEmployeesValue()` path in [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:11). That function now assumes every imported `employees` cell is JSON with the exact shape `{ min, max }`, then immediately does `parsed.min.toString()` and `parsed.max.toString()`.

The rest of the app does not use that shape. `employees` is stored as a nullable `String` in Prisma ([prisma/schema.prisma](/home/user/nextcrm-app-research/prisma/schema.prisma:1236)), the create/update target forms treat it as free text with placeholder `50-200` ([NewTargetForm.tsx](/home/user/nextcrm-app-research/app/[locale]/(routes)/campaigns/targets/components/NewTargetForm.tsx:311), [UpdateTargetForm.tsx](/home/user/nextcrm-app-research/app/[locale]/(routes)/campaigns/targets/components/UpdateTargetForm.tsx:347)), and e2e coverage also enters `50-200` as a plain string ([campaign-targets.spec.ts](/home/user/nextcrm-app-research/tests/e2e/campaign-targets.spec.ts:73)).

**Why This Produced This Exact Error**

The stack trace is `Cannot read properties of undefined (reading 'toString')` at `parsed.min.toString()`, not a JSON parse error. That means the CSV cell was likely valid JSON, but not an object containing both `min` and `max`. Examples that would produce this exact failure are a JSON string like `"50-200"` or an object missing one of those keys. In other words, the importer accepted a value shape that `JSON.parse()` can read, but `parseEmployeesValue()` cannot safely dereference.

**Contributing Factors**

`parseEmployeesValue(row.employees)` is executed for every row before row validation and before any row can be skipped ([import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:41)). So one malformed `employees` value causes the entire import request to fail with HTTP 500 instead of skipping a row or preserving the original string.

The preview flow in [components/modals/ImportTargetsModal.tsx](/home/user/nextcrm-app-research/components/modals/ImportTargetsModal.tsx:140) only checks for missing `last_name` or `company`; it never validates `employees`. That allowed the bad file to pass preview and fail only at server import time.

**Conclusion**

The most likely root cause is a bad assumption added in the latest change: the importer now requires `employees` CSV values to be structured JSON `{ min, max }`, while actual application behavior and likely CSV inputs use plain strings or other legacy JSON shapes. That mismatch is what triggered the 2026-04-12 00:18:46Z incident.

function normalizeCellValue(value?: string) {
  return typeof value === "string" ? value.trim() : "";
}

function parseEmployeesValue(value?: string) {
  const parsed = JSON.parse(normalizeCellValue(value) || "null") as {
    min: number;
    max: number;
  };

  return `${parsed.min.toString()}-${parsed.max.toString()}`;
}

**Most Likely Root Cause**

The import regression was introduced by the new `parseEmployeesValue()` path in [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:11). That function now assumes every imported `employees` cell is JSON with the exact shape `{ min, max }`, then immediately does `parsed.min.toString()` and `parsed.max.toString()`.

The rest of the app does not use that shape. `employees` is stored as a nullable `String` in Prisma ([prisma/schema.prisma](/home/user/nextcrm-app-research/prisma/schema.prisma:1236)), the create/update target forms treat it as free text with placeholder `50-200` ([NewTargetForm.tsx](/home/user/nextcrm-app-research/app/[locale]/(routes)/campaigns/targets/components/NewTargetForm.tsx:311), [UpdateTargetForm.tsx](/home/user/nextcrm-app-research/app/[locale]/(routes)/campaigns/targets/components/UpdateTargetForm.tsx:347)), and e2e coverage also enters `50-200` as a plain string ([campaign-targets.spec.ts](/home/user/nextcrm-app-research/tests/e2e/campaign-targets.spec.ts:73)).

**Why This Produced This Exact Error**

The stack trace is `Cannot read properties of undefined (reading 'toString')` at `parsed.min.toString()`, not a JSON parse error. That means the CSV cell was likely valid JSON, but not an object containing both `min` and `max`. Examples that would produce this exact failure are a JSON string like `"50-200"` or an object missing one of those keys. In other words, the importer accepted a value shape that `JSON.parse()` can read, but `parseEmployeesValue()` cannot safely dereference.

**Contributing Factors**

`parseEmployeesValue(row.employees)` is executed for every row before row validation and before any row can be skipped ([import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:41)). So one malformed `employees` value causes the entire import request to fail with HTTP 500 instead of skipping a row or preserving the original string.

The preview flow in [components/modals/ImportTargetsModal.tsx](/home/user/nextcrm-app-research/components/modals/ImportTargetsModal.tsx:140) only checks for missing `last_name` or `company`; it never validates `employees`. That allowed the bad file to pass preview and fail only at server import time.

**Conclusion**

The most likely root cause is a bad assumption added in the latest change: the importer now requires `employees` CSV values to be structured JSON `{ min, max }`, while actual application behavior and likely CSV inputs use plain strings or other legacy JSON shapes. That mismatch is what triggered the 2026-04-12 00:18:46Z incident.
