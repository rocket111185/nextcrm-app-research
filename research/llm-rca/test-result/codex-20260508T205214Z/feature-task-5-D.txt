**Incident Report**

Most likely root cause: `parseEmployeesValue()` in `actions/crm/targets/import-targets.ts` throws during CSV target import when the `employees` value is missing, empty, or not strict JSON.

The failure was introduced in commit `9790f4c1` by this new path:

```ts
const employees = parseEmployeesValue(row.employees);
```

`parseEmployeesValue()` does:

```ts
const parsed = JSON.parse(normalizeCellValue(value) || "null");
return `${parsed.min.toString()}-${parsed.max.toString()}`;
```

That means common CSV inputs fail before import reaches the database:

- missing/skipped `employees` column -> parses `"null"` -> `parsed.min` throws `TypeError`
- blank employees cell -> same `TypeError`
- existing range like `50-200` -> `JSON.parse()` throws `SyntaxError`
- plain number like `50` -> parses as number, then `parsed.min` is invalid

The client preview does not validate or transform `employees`; it only checks `last_name` / `company`, so users can reach the final Import button with rows that will crash server-side.

The schema also confirms `crm_Targets.employees` is just `String?`, and the existing target form uses a free-text placeholder like `50-200`, so requiring a JSON object `{ min, max }` during import is inconsistent with the rest of the app.

The observability message `"Unhandled request error"` matches this: the exception occurs inside the server action and is picked up by Next’s `onRequestError` instrumentation. The null `error.name/message/stack` in the incident payload appears to be a logging/serialization limitation, not evidence that the thrown value was actually null.

No code changes were made.

**Incident Report**

Most likely root cause: `parseEmployeesValue()` in `actions/crm/targets/import-targets.ts` throws during CSV target import when the `employees` value is missing, empty, or not strict JSON.

The failure was introduced in commit `9790f4c1` by this new path:

```ts
const employees = parseEmployeesValue(row.employees);
```

`parseEmployeesValue()` does:

```ts
const parsed = JSON.parse(normalizeCellValue(value) || "null");
return `${parsed.min.toString()}-${parsed.max.toString()}`;
```

That means common CSV inputs fail before import reaches the database:

- missing/skipped `employees` column -> parses `"null"` -> `parsed.min` throws `TypeError`
- blank employees cell -> same `TypeError`
- existing range like `50-200` -> `JSON.parse()` throws `SyntaxError`
- plain number like `50` -> parses as number, then `parsed.min` is invalid

The client preview does not validate or transform `employees`; it only checks `last_name` / `company`, so users can reach the final Import button with rows that will crash server-side.

The schema also confirms `crm_Targets.employees` is just `String?`, and the existing target form uses a free-text placeholder like `50-200`, so requiring a JSON object `{ min, max }` during import is inconsistent with the rest of the app.

The observability message `"Unhandled request error"` matches this: the exception occurs inside the server action and is picked up by Next’s `onRequestError` instrumentation. The null `error.name/message/stack` in the incident payload appears to be a logging/serialization limitation, not evidence that the thrown value was actually null.

No code changes were made.
