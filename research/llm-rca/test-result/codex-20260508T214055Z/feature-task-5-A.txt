**Incident Report**

Most likely root cause: the CSV target import path assumes the `employees` field is always a JSON object with numeric `min` and `max` properties, but the imported CSV likely supplied a normal employee/headcount value such as `150`, `"150"`, `{}`, or another JSON-parseable value without `min`/`max`.

Evidence:

- The stack points directly to `parseEmployeesValue()` inside `importTargets()`.
- In [actions/crm/targets/import-targets.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/actions/crm/targets/import-targets.ts:11), the parser does:

```ts
const parsed = JSON.parse(normalizeCellValue(value) || "null");
return `${parsed.min.toString()}-${parsed.max.toString()}`;
```

- The observed error is:

```text
Cannot read properties of undefined (reading 'toString')
```

That means `parsed.min` or `parsed.max` was `undefined`. This is consistent with an employee value that parsed successfully but was not shaped like `{ min: number, max: number }`.

- `importTargets()` calls `parseEmployeesValue(row.employees)` unconditionally before validating the row, so any malformed/mismatched employee value aborts the whole server action.
- The import UI includes `employees` as a mappable field, and `suggestMapping()` aliases headers like `headcount`, `numemployees`, `teamsize`, and `companysize` to `employees`. That makes it likely a standard CSV headcount column was automatically mapped to `employees`.
- The preview step only validates `last_name` or `company`; it does not parse or validate `employees`, so the user can reach import successfully and then hit the server-side crash.

Likely regression source: commit `9790f4c` (`Handle legacy employee range imports`) introduced `parseEmployeesValue()` and changed imported `employees` from `row.employees || null` to the parsed range format. The implementation handles only one legacy structured format and does not tolerate ordinary CSV values.

Impact: POST `/en/campaigns/targets` fails during CSV import when an `employees` column is mapped and contains anything other than a strict JSON object with both `min` and `max`. The error is unhandled, so the import aborts instead of skipping the bad row or importing the raw value.

function normalizeCellValue(value?: string) {
  return typeof value === "string" ? value.trim() : "";

function parseEmployeesValue(value?: string) {
  const parsed = JSON.parse(normalizeCellValue(value) || "null") as {
    min: number;
    max: number;
  };

  return `${parsed.min.toString()}-${parsed.max.toString()}`;
}

Likely regression source: commit `9790f4c` (`Handle legacy employee range imports`) introduced `parseEmployeesValue()` and changed imported `employees` from `row.employees || null` to the parsed range format. The implementation handles only one legacy structured format and does not tolerate ordinary CSV values.

Impact: POST `/en/campaigns/targets` fails during CSV import when an `employees` column is mapped and contains anything other than a strict JSON object with both `min` and `max`. The error is unhandled, so the import aborts instead of skipping the bad row or importing the raw value.
