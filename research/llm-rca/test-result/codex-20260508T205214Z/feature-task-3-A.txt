Most likely root cause: `NEXT_PUBLIC_MINIO_ENDPOINT` is configured as an empty string in the local environment, and the upload presigned URL endpoint passes it directly into `new URL()`.

Evidence:

- Error: `ERR_INVALID_URL`, `err_input: ""`
- Failing function: `buildPublicFileUrl`
- Source: `app/api/upload/presigned-url/route.ts`
- Code path:

```ts
const publicBaseUrl = new URL(MINIO_PUBLIC_URL!);
```

- `MINIO_PUBLIC_URL` comes from `lib/minio.ts`:

```ts
export const MINIO_PUBLIC_URL = process.env.NEXT_PUBLIC_MINIO_ENDPOINT;
```

- Local config has:

```env
NEXT_PUBLIC_MINIO_ENDPOINT=
```

The request itself appears valid: `contentType` is `application/pdf`, and `folder` is `documents`, both accepted by the route. The failure occurs after presigned URL generation begins, when the API tries to construct the public file URL to return to the client.

Conclusion: the incident was caused by missing local MinIO public endpoint configuration, specifically empty `NEXT_PUBLIC_MINIO_ENDPOINT`. The expected value should be a valid absolute URL such as `https://your-minio-host` or the local MinIO public URL.
