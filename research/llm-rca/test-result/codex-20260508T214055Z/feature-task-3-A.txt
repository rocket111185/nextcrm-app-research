Most likely root cause: `NEXT_PUBLIC_MINIO_ENDPOINT` is configured as an empty string in the local environment, so `/api/upload/presigned-url` crashes while building the returned public file URL.

Evidence:
- The incident error is `ERR_INVALID_URL` with `err_input: ""`.
- The stack points directly to `buildPublicFileUrl`.
- In [app/api/upload/presigned-url/route.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/app/api/upload/presigned-url/route.ts:15), `buildPublicFileUrl()` calls `new URL(MINIO_PUBLIC_URL!)`.
- In [lib/minio.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/lib/minio.ts:19), `MINIO_PUBLIC_URL` comes from `process.env.NEXT_PUBLIC_MINIO_ENDPOINT`.
- `.env.local` has `NEXT_PUBLIC_MINIO_ENDPOINT=` at line 77, and `.env` also has it empty.

The upload request itself appears valid: `contentType` is `application/pdf`, `folder` is `documents`, both are allowed by the route, and the user was authenticated. The failure happens after the presigned URL is requested, when the server tries to construct `fileUrl` for the response.

Contributing factor: `lib/minio.ts` validates `MINIO_ENDPOINT`, `MINIO_ACCESS_KEY`, `MINIO_SECRET_KEY`, and `MINIO_BUCKET`, but does not validate `NEXT_PUBLIC_MINIO_ENDPOINT`, even though the upload route requires it.

Recommended fix direction: set `NEXT_PUBLIC_MINIO_ENDPOINT` to the externally reachable MinIO/S3 base URL, for example `http://localhost:9000` or the deployed storage endpoint, and restart the Next.js dev server so the environment is reloaded.

Most likely root cause: `NEXT_PUBLIC_MINIO_ENDPOINT` is configured as an empty string in the local environment, so `/api/upload/presigned-url` crashes while building the returned public file URL.

Evidence:
- The incident error is `ERR_INVALID_URL` with `err_input: ""`.
- The stack points directly to `buildPublicFileUrl`.
- In [app/api/upload/presigned-url/route.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/app/api/upload/presigned-url/route.ts:15), `buildPublicFileUrl()` calls `new URL(MINIO_PUBLIC_URL!)`.

- `.env.local` has `NEXT_PUBLIC_MINIO_ENDPOINT=` at line 77, and `.env` also has it empty.

The upload request itself appears valid: `contentType` is `application/pdf`, `folder` is `documents`, both are allowed by the route, and the user was authenticated. The failure happens after the presigned URL is requested, when the server tries to construct `fileUrl` for the response.

Contributing factor: `lib/minio.ts` validates `MINIO_ENDPOINT`, `MINIO_ACCESS_KEY`, `MINIO_SECRET_KEY`, and `MINIO_BUCKET`, but does not validate `NEXT_PUBLIC_MINIO_ENDPOINT`, even though the upload route requires it.

Recommended fix direction: set `NEXT_PUBLIC_MINIO_ENDPOINT` to the externally reachable MinIO/S3 base URL, for example `http://localhost:9000` or the deployed storage endpoint, and restart the Next.js dev server so the environment is reloaded.

Most likely root cause: `NEXT_PUBLIC_MINIO_ENDPOINT` is configured as an empty string in the local environment, so `/api/upload/presigned-url` crashes while building the returned public file URL.

Evidence:
- The incident error is `ERR_INVALID_URL` with `err_input: ""`.
- The stack points directly to `buildPublicFileUrl`.
- In [app/api/upload/presigned-url/route.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/app/api/upload/presigned-url/route.ts:15), `buildPublicFileUrl()` calls `new URL(MINIO_PUBLIC_URL!)`.
- In [lib/minio.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/lib/minio.ts:19), `MINIO_PUBLIC_URL` comes from `process.env.NEXT_PUBLIC_MINIO_ENDPOINT`.
- `.env.local` has `NEXT_PUBLIC_MINIO_ENDPOINT=` at line 77, and `.env` also has it empty.

The upload request itself appears valid: `contentType` is `application/pdf`, `folder` is `documents`, both are allowed by the route, and the user was authenticated. The failure happens after the presigned URL is requested, when the server tries to construct `fileUrl` for the response.

Contributing factor: `lib/minio.ts` validates `MINIO_ENDPOINT`, `MINIO_ACCESS_KEY`, `MINIO_SECRET_KEY`, and `MINIO_BUCKET`, but does not validate `NEXT_PUBLIC_MINIO_ENDPOINT`, even though the upload route requires it.

Recommended fix direction: set `NEXT_PUBLIC_MINIO_ENDPOINT` to the externally reachable MinIO/S3 base URL, for example `http://localhost:9000` or the deployed storage endpoint, and restart the Next.js dev server so the environment is reloaded.
