function normalizeCellValue(value?: string) {
  return typeof value === "string" ? value.trim() : "";
}

function parseEmployeesValue(value?: string) {
  const parsed = JSON.parse(normalizeCellValue(value) || "null") as {
    min: number;
    max: number;
  };

  return `${parsed.min.toString()}-${parsed.max.toString()}`;
}

**Most likely root cause**

A regression was introduced in commit `234a54c` (`Handle legacy employee range imports`) on **April 12, 2026 at 03:05:10 +03:00**. The incident happened on **April 12, 2026 at 00:18:46Z** = **03:18:46 +03:00**, about **13 minutes later**.

That commit added `parseEmployeesValue()` in [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:11) and made every imported row call it unconditionally at [line 59](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:59). The function assumes the CSV `employees` cell is JSON shaped exactly like `{ min, max }`, then does:

- `parsed.min.toString()`
- `parsed.max.toString()`

The observed error means `JSON.parse(...)` returned a value where `min` or `max` was missing, so `parsed.min` or `parsed.max` was `undefined`, and `.toString()` crashed.

**Why this is the likely regression**

- Before that commit, the importer passed `row.employees` through as plain text.
- Elsewhere in the app, `employees` is still treated as a normal string field, not a structured JSON object.
- The UI keeps `employees` optional in [ImportTargetsModal.tsx](/home/user/nextcrm-app-research/components/modals/ImportTargetsModal.tsx:56).
- Existing test coverage also uses plain text like `50-200` in [tests/e2e/campaign-targets.spec.ts](/home/user/nextcrm-app-research/tests/e2e/campaign-targets.spec.ts:73).

So the importer now enforces a stricter format than the rest of the product expects, with no validation or fallback.

**Contributing factors**

- `parseEmployeesValue()` has no guard for missing/partial/malformed structures.
- `importTargets()` runs that parser before row validation and before building the insert payload.
- The modal change only added explanatory text; it does not validate or normalize `employees` into the required structure.

**Conclusion**

The incident was most likely caused by the April 12 importer change that unconditionally re-parses `employees` as `{ min, max }` JSON. A CSV row contained an `employees` value that parsed successfully but did not include both properties, causing the `undefined.toString()` crash and the 500 on `POST /en/campaigns/targets`.
