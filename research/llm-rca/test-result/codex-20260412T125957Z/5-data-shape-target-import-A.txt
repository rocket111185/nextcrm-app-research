**Most Likely Root Cause**

A regression was introduced on **April 12, 2026** in commit `234a54c` (`Handle legacy employee range imports`). That change made the CSV importer unconditionally parse the `employees` column as JSON and then immediately access `parsed.min` / `parsed.max` in [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:11).

The crash you saw:

`Cannot read properties of undefined (reading 'toString')`

matches the code at [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:17), which means `JSON.parse(...)` returned a value that did not contain `min` and/or `max`. In other words, the importer received an `employees` cell that was valid JSON, but not in the exact shape `{ "min": number, "max": number }`.

**Why this is the likeliest cause**

- The importer now always runs `parseEmployeesValue(row.employees)` for every row at [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:59).
- `parseEmployeesValue` has no guards for:
  - empty values
  - plain strings like `50-200`
  - single numbers like `50`
  - alternate JSON like `{}` or `{ "label": "50-200" }`
  - partial JSON like `{ "min": 50 }`
- Elsewhere in the app, `employees` is treated as a plain string, not structured JSON:
  - manual create form placeholder is `50-200` in [NewTargetForm.tsx](/home/user/nextcrm-app-research/app/[locale]/(routes)/campaigns/targets/components/NewTargetForm.tsx:311)
  - manual update form uses the same plain string input in [UpdateTargetForm.tsx](/home/user/nextcrm-app-research/app/[locale]/(routes)/campaigns/targets/components/UpdateTargetForm.tsx:347)
  - account reporting parses `employees` as a numeric/string field in [actions/reports/accounts.ts](/home/user/nextcrm-app-research/actions/reports/accounts.ts:57)

**Conclusion**

The incident is most likely caused by a **format-contract mismatch introduced by the new legacy import handling**: the importer assumes all CSV `employees` values are JSON `{min,max}`, but real imported data is not consistently in that format. The first row with an incompatible but JSON-parsable value causes the 500 before row-level validation can skip or normalize it.

**Contributing factors**

- No validation around the parsed object before calling `.toString()`
- No fallback to the existing plain-string format used everywhere else
- The import path fails hard on one bad `employees` value instead of recording a row error and continuing

function normalizeCellValue(value?: string) {
  return typeof value === "string" ? value.trim() : "";

function parseEmployeesValue(value?: string) {
  const parsed = JSON.parse(normalizeCellValue(value) || "null") as {
    min: number;
    max: number;
  };

  return `${parsed.min.toString()}-${parsed.max.toString()}`;
}

**Most Likely Root Cause**

A regression was introduced on **April 12, 2026** in commit `234a54c` (`Handle legacy employee range imports`). That change made the CSV importer unconditionally parse the `employees` column as JSON and then immediately access `parsed.min` / `parsed.max` in [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:11).

The crash you saw:

`Cannot read properties of undefined (reading 'toString')`

matches the code at [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:17), which means `JSON.parse(...)` returned a value that did not contain `min` and/or `max`. In other words, the importer received an `employees` cell that was valid JSON, but not in the exact shape `{ "min": number, "max": number }`.

**Why this is the likeliest cause**

- The importer now always runs `parseEmployeesValue(row.employees)` for every row at [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:59).
- `parseEmployeesValue` has no guards for:
  - empty values
  - plain strings like `50-200`
  - single numbers like `50`
  - alternate JSON like `{}` or `{ "label": "50-200" }`
  - partial JSON like `{ "min": 50 }`
- Elsewhere in the app, `employees` is treated as a plain string, not structured JSON:
  - manual create form placeholder is `50-200` in [NewTargetForm.tsx](/home/user/nextcrm-app-research/app/[locale]/(routes)/campaigns/targets/components/NewTargetForm.tsx:311)
  - manual update form uses the same plain string input in [UpdateTargetForm.tsx](/home/user/nextcrm-app-research/app/[locale]/(routes)/campaigns/targets/components/UpdateTargetForm.tsx:347)
  - account reporting parses `employees` as a numeric/string field in [actions/reports/accounts.ts](/home/user/nextcrm-app-research/actions/reports/accounts.ts:57)

**Conclusion**

The incident is most likely caused by a **format-contract mismatch introduced by the new legacy import handling**: the importer assumes all CSV `employees` values are JSON `{min,max}`, but real imported data is not consistently in that format. The first row with an incompatible but JSON-parsable value causes the 500 before row-level validation can skip or normalize it.

**Contributing factors**

- No validation around the parsed object before calling `.toString()`
- No fallback to the existing plain-string format used everywhere else
- The import path fails hard on one bad `employees` value instead of recording a row error and continuing
