`NEXT_PUBLIC_MINIO_ENDPOINT` is empty in the active local environment, and the upload route uses it without validation.

**Why this is the likely cause**

The failure happens at:

`app/api/upload/presigned-url/route.ts:13`
```ts
const publicBaseUrl = new URL(MINIO_PUBLIC_URL!);
```

`MINIO_PUBLIC_URL` comes from:

`lib/minio.ts`
```ts
export const MINIO_PUBLIC_URL = process.env.NEXT_PUBLIC_MINIO_ENDPOINT;
```

The incident stack already shows the exact bad input:

`code: 'ERR_INVALID_URL', input: ''`

That means `new URL(...)` received an empty string, not just a malformed hostname.

**Repository evidence**

`/.env.local` currently contains:
```env
NEXT_PUBLIC_MINIO_ENDPOINT=
MINIO_ENDPOINT="https://your-minio-host"
MINIO_BUCKET="your-bucket-name"
```

So the server-side MinIO client can still be configured well enough to reach the presign flow, but the separate public URL base is blank. The route then fails after presigned URL generation when constructing the returned `fileUrl`.

**Secondary contributing factor**

There is startup validation for `MINIO_ENDPOINT`, `MINIO_ACCESS_KEY`, `MINIO_SECRET_KEY`, and `MINIO_BUCKET`, but not for `NEXT_PUBLIC_MINIO_ENDPOINT`. Because of that, the app starts successfully and only fails at request time in `/api/upload/presigned-url`.

**Conclusion**

The most likely root cause is a misconfigured environment: `NEXT_PUBLIC_MINIO_ENDPOINT` is unset/empty in `.env.local`, causing `buildPublicFileUrl()` to throw `TypeError: Invalid URL`.
