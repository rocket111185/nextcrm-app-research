===== RESPONSE =====
**Most Likely Root Cause**

The import path for campaign targets crashes when processing the `employees` CSV field.

The failing code is in `actions/crm/targets/import-targets.ts:11-17`:

```ts
function parseEmployeesValue(value?: string) {
  const parsed = JSON.parse(normalizeCellValue(value) || "null") as {
    min: number;
    max: number;
  };

  return `${parsed.min.toString()}-${parsed.max.toString()}`;
}
```

The incident stack points directly to this function during `importTargets()` for `POST /en/campaigns/targets`. The observed error, `Cannot read properties of undefined (reading 'toString')`, means `parsed.min` was undefined when `.toString()` was called.

**Why This Happened**

`employees` is modeled elsewhere as a plain optional string:

- Prisma: `crm_Targets.employees String?`
- New/edit forms use free text with placeholder `50-200`
- Existing planning docs show import should store `row.employees || null`

But the importer assumes the CSV value is JSON shaped like:

```json
{ "min": 50, "max": 200 }
```

That assumption does not match the rest of the app. A CSV value such as `100`, `{}`, or another JSON value without `min` will parse successfully but leave `parsed.min` undefined, producing the exact incident error.

**Trigger**

A user imported a CSV from `/en/campaigns/targets` where the mapped `employees` column contained a value that was not a JSON object with both `min` and `max`.

**Contributing Issue**

The preview step only checks whether each row has `last_name` or `company`. It does not validate or normalize `employees`, so the user can reach the final import action with data that will crash server-side.

**Confidence**

High. The stack trace, compiled server output, and source code all converge on `parseEmployeesValue(row.employees)` inside the target CSV import action. The application’s schema and forms indicate `employees` should be treated as a string, not parsed as a JSON range object.

function normalizeCellValue(value?: string) {
  return typeof value === "string" ? value.trim() : "";

function parseEmployeesValue(value?: string) {
  const parsed = JSON.parse(normalizeCellValue(value) || "null") as {
    min: number;
    max: number;
  };

  return `${parsed.min.toString()}-${parsed.max.toString()}`;
}

**Most Likely Root Cause**

The import path for campaign targets crashes when processing the `employees` CSV field.

The failing code is in `actions/crm/targets/import-targets.ts:11-17`:

```ts
function parseEmployeesValue(value?: string) {
  const parsed = JSON.parse(normalizeCellValue(value) || "null") as {
    min: number;
    max: number;
  };

  return `${parsed.min.toString()}-${parsed.max.toString()}`;
}
```

The incident stack points directly to this function during `importTargets()` for `POST /en/campaigns/targets`. The observed error, `Cannot read properties of undefined (reading 'toString')`, means `parsed.min` was undefined when `.toString()` was called.

**Why This Happened**

`employees` is modeled elsewhere as a plain optional string:

- Prisma: `crm_Targets.employees String?`
- New/edit forms use free text with placeholder `50-200`
- Existing planning docs show import should store `row.employees || null`

But the importer assumes the CSV value is JSON shaped like:

```json
{ "min": 50, "max": 200 }
```

That assumption does not match the rest of the app. A CSV value such as `100`, `{}`, or another JSON value without `min` will parse successfully but leave `parsed.min` undefined, producing the exact incident error.

**Trigger**

A user imported a CSV from `/en/campaigns/targets` where the mapped `employees` column contained a value that was not a JSON object with both `min` and `max`.

**Contributing Issue**

The preview step only checks whether each row has `last_name` or `company`. It does not validate or normalize `employees`, so the user can reach the final import action with data that will crash server-side.

**Confidence**

High. The stack trace, compiled server output, and source code all converge on `parseEmployeesValue(row.employees)` inside the target CSV import action. The application’s schema and forms indicate `employees` should be treated as a string, not parsed as a JSON range object.
