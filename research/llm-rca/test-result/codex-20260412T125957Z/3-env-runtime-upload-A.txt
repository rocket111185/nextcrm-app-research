- The exception is raised at [app/api/upload/presigned-url/route.ts](/home/user/nextcrm-app-research/app/api/upload/presigned-url/route.ts:13), where `buildPublicFileUrl()` does `new URL(MINIO_PUBLIC_URL!)`.
- `MINIO_PUBLIC_URL` is just `process.env.NEXT_PUBLIC_MINIO_ENDPOINT` in [lib/minio.ts](/home/user/nextcrm-app-research/lib/minio.ts:19).
- The stack trace includes `input: ''`, which means the env var was not merely missing in code flow; it was present as an empty string when `new URL()` ran.
- The repo’s checked-in env files show exactly that override pattern:
  - [.env](/home/user/nextcrm-app-research/.env:32) sets `NEXT_PUBLIC_MINIO_ENDPOINT="https://your-minio-host"`
  - [.env.local](/home/user/nextcrm-app-research/.env.local:77) sets `NEXT_PUBLIC_MINIO_ENDPOINT=`
  - [.env.local.example](/home/user/nextcrm-app-research/.env.local.example:77) also leaves it blank

**Why This Is the Most Likely Cause**

In Next.js, `.env.local` has higher precedence than `.env`. So if the running environment matches this repo setup, the empty `NEXT_PUBLIC_MINIO_ENDPOINT=` in `.env.local` overrides the valid value from `.env`, producing the exact observed failure.

**Contributing Issue**

The code validates `MINIO_ENDPOINT`, `MINIO_ACCESS_KEY`, `MINIO_SECRET_KEY`, and `MINIO_BUCKET` at module load, but does not validate `NEXT_PUBLIC_MINIO_ENDPOINT` in [lib/minio.ts](/home/user/nextcrm-app-research/lib/minio.ts:3). That allowed a bad config to survive until request time and fail as a 500 instead of failing fast with a clearer startup error.

**Conclusion**

The incident is most likely caused by a misconfigured environment: `NEXT_PUBLIC_MINIO_ENDPOINT` is blank in the effective runtime config, most plausibly due to `.env.local` overriding `.env`.
