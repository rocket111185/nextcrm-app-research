**Most Likely Root Cause**

The 500 on `POST /en/campaigns/targets` was introduced by the new `parseEmployeesValue()` helper in [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:11). That helper now assumes every imported `employees` cell is valid JSON with this exact shape:

```ts
{ min: number; max: number }
```

It then immediately does:

```ts
return `${parsed.min.toString()}-${parsed.max.toString()}`;
```

The crash means `JSON.parse(...)` returned something that did not contain `min` and/or `max`, so `parsed.min` was `undefined` and `.toString()` threw. This matches the stack exactly at line 17.

**Why this is the likely regression**

The previous implementation stored `row.employees` directly and did not parse it at all. The failing behavior appeared only after the last change set, which added `parseEmployeesValue()` and switched the import path to use it. The rest of the codebase still treats `employees` as a plain string field, not a structured object:

- Manual create/update forms use a normal text input with placeholder `50-200`.
- E2E coverage fills `Employees` with the plain string `50-200`.
- Prisma stores `employees` as `String?`.

So the server-side import path became stricter than the rest of the app and is now incompatible with normal or legacy CSV content.

**Contributing factors**

- `parseEmployeesValue()` runs before row validation, so one bad `employees` value aborts the entire import instead of skipping a row.
- There is no shape check or fallback for empty cells, plain strings, or alternate JSON formats.
- The modal text added in the same change mentions “legacy imports” expecting structured employee ranges, but there is no corresponding enforcement or conversion on the client side; the server still receives raw CSV strings.

**Conclusion**

The incident was most likely caused by a regression in the April 12, 2026 import change: `employees` import values were changed from pass-through strings to mandatory JSON `{min,max}` parsing, but actual CSV data still contains non-matching values.

function normalizeCellValue(value?: string) {
  return typeof value === "string" ? value.trim() : "";
}

export async function importTargets(
  formData: FormData
): Promise<{ imported: number; skipped: number; errors: string[] }> {
  const session = await getSession();
  if (!session) throw new Error("Unauthorized");

  const file = formData.get("file") as File | null;
  if (!file) throw new Error("No file provided");

  const mappingRaw = formData.get("mapping") as string | null;
  const mapping: Record<string, string> = mappingRaw ? JSON.parse(mappingRaw) : {};

  const text = await file.text();
  const { data } = Papa.parse<Record<string, string>>(text, {
    header: true,
    skipEmptyLines: true,
  });

  const valid: any[] = [];
  const errors: string[] = [];

  data.forEach((rawRow, index) => {
    const row: Record<string, string> = {};
    if (Object.keys(mapping).length > 0) {
      Object.entries(mapping).forEach(([csvCol, targetField]) => {
        if (targetField && rawRow[csvCol] !== undefined) {
          row[targetField] = rawRow[csvCol];
        }
      });
    } else {
      Object.assign(row, rawRow);
    }

    const last_name = normalizeCellValue(row.last_name);
    const email = normalizeCellValue(row.email).toLowerCase();
    const mobile_phone = normalizeCellValue(row.mobile_phone);
    const company = row.company?.trim();
    const company_website = normalizeCellValue(row.company_website);
    const companyDomain = company_website || company.trim().toLowerCase();

    if (!last_name && !company) {
      errors.push(`Row ${index + 2}: missing last_name or company`);
      return;
    }

    valid.push({
      last_name: last_name ?? "",
      first_name: row.first_name || null,
      email: email || null,
      mobile_phone: mobile_phone || null,
      office_phone: row.office_phone || null,
      company: row.company || null,
      position: row.position || null,
      company_website: companyDomain || null,
      personal_website: row.personal_website || null,
      social_linkedin: row.social_linkedin || null,
      social_x: row.social_x || null,
      social_instagram: row.social_instagram || null,
      social_facebook: row.social_facebook || null,
      personal_email: row.personal_email || null,
      company_email: row.company_email || null,
      company_phone: row.company_phone || null,
      city: row.city || null,
      country: row.country || null,
      industry: row.industry || null,
      employees: row.employees || null,
      description: row.description || null,
      created_by: (session.user as any).id,
    });
  });

  if (valid.length > 0) {
    await prismadb.crm_Targets.createMany({ data: valid, skipDuplicates: true });
  }

  return { imported: valid.length, skipped: errors.length, errors };
}

**Most Likely Root Cause**

The 500 on `POST /en/campaigns/targets` was introduced by the new `parseEmployeesValue()` helper in [actions/crm/targets/import-targets.ts](/home/user/nextcrm-app-research/actions/crm/targets/import-targets.ts:11). That helper now assumes every imported `employees` cell is valid JSON with this exact shape:

```ts
{ min: number; max: number }
```

It then immediately does:

```ts
return `${parsed.min.toString()}-${parsed.max.toString()}`;
```

The crash means `JSON.parse(...)` returned something that did not contain `min` and/or `max`, so `parsed.min` was `undefined` and `.toString()` threw. This matches the stack exactly at line 17.

**Why this is the likely regression**

The previous implementation stored `row.employees` directly and did not parse it at all. The failing behavior appeared only after the last change set, which added `parseEmployeesValue()` and switched the import path to use it. The rest of the codebase still treats `employees` as a plain string field, not a structured object:

- Manual create/update forms use a normal text input with placeholder `50-200`.
- E2E coverage fills `Employees` with the plain string `50-200`.
- Prisma stores `employees` as `String?`.

So the server-side import path became stricter than the rest of the app and is now incompatible with normal or legacy CSV content.

**Contributing factors**

- `parseEmployeesValue()` runs before row validation, so one bad `employees` value aborts the entire import instead of skipping a row.
- There is no shape check or fallback for empty cells, plain strings, or alternate JSON formats.
- The modal text added in the same change mentions “legacy imports” expecting structured employee ranges, but there is no corresponding enforcement or conversion on the client side; the server still receives raw CSV strings.

**Conclusion**

The incident was most likely caused by a regression in the April 12, 2026 import change: `employees` import values were changed from pass-through strings to mandatory JSON `{min,max}` parsing, but actual CSV data still contains non-matching values.
