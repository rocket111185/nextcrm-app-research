`parseEmployeesValue()` does this unconditionally:

```ts
const parsed = JSON.parse(normalizeCellValue(value) || "null");
return `${parsed.min.toString()}-${parsed.max.toString()}`;
```

If the CSV value is a normal employee count like `100`, `"100"`, `{}`, or any object without `min`/`max`, `parsed.min` or `parsed.max` is `undefined`, causing the observed `toString()` crash. The import code calls this before validating/skipping the row, so one malformed or ordinary employee value aborts the whole server action.

function send(level: ClientLogLevel, message: string, context: ClientLogContext) {
  if (typeof window === "undefined") {
    return;
  }

  const payload = JSON.stringify({
    level,
    message,
    context: serialize(context),
    sessionId,
    url: window.location.href,
    pathname: window.location.pathname,
    timestamp: new Date().toISOString(),
  });

  if (navigator.sendBeacon) {
    const sent = navigator.sendBeacon(
      ENDPOINT,
      new Blob([payload], { type: "application/json" })
    );
    if (sent) {
      return;
    }
  }

  fetch(ENDPOINT, {
    method: "POST",
    headers: { "content-type": "application/json" },
    body: payload,
    keepalive: true,
  }).catch(() => {
    // Client logging must never break the UI.
  });
}

export function createClientLogger(bindings: ClientLogContext) {
  const log = (level: ClientLogLevel, context: ClientLogContext, message: string) => {
    send(level, message, { ...bindings, ...context });

    debug: (context: ClientLogContext, message: string) => log("debug", context, message),
    info: (context: ClientLogContext, message: string) => log("info", context, message),
    warn: (context: ClientLogContext, message: string) => log("warn", context, message),
    error: (context: ClientLogContext, message: string) => log("error", context, message),
  };
}

The raw local log for trace `b7fc50998917f6d53bb5de0ae8776eb5` shows the actual error hidden by the incident summary:

`TypeError: Cannot read properties of undefined (reading 'toString')`

It occurred during:

`POST /en/campaigns/targets`  
route: `/[locale]/campaigns/targets`  
route type: `action`  
function path: `importTargets -> parseEmployeesValue`

Relevant code: [actions/crm/targets/import-targets.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/actions/crm/targets/import-targets.ts:11)

`parseEmployeesValue()` does this unconditionally:

```ts
const parsed = JSON.parse(normalizeCellValue(value) || "null");
return `${parsed.min.toString()}-${parsed.max.toString()}`;
```

If the CSV value is a normal employee count like `100`, `"100"`, `{}`, or any object without `min`/`max`, `parsed.min` or `parsed.max` is `undefined`, causing the observed `toString()` crash. The import code calls this before validating/skipping the row, so one malformed or ordinary employee value aborts the whole server action.

A nearby warning, `API key not configured`, likely caused mapping suggestions to fall back to fuzzy matching. That may have mapped a CSV header like `headcount`, `numemployees`, `team size`, or `company size` to `employees`, but it is not the direct failure. The direct failure is the importer’s overly strict and unguarded parser for the `employees` field.
