**Incident Report**

Most likely root cause: `actions/crm/targets/import-targets.ts` introduced an unsafe employee-range parser that assumes every imported `employees` value is JSON shaped as `{ "min": number, "max": number }`.

The crash occurs here: [import-targets.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/actions/crm/targets/import-targets.ts:11)

```ts
const parsed = JSON.parse(normalizeCellValue(value) || "null");
return `${parsed.min.toString()}-${parsed.max.toString()}`;
```

The server action calls this unconditionally for each CSV row at [import-targets.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/actions/crm/targets/import-targets.ts:59), before row validation completes.

**Why this matches the incident**

The observed error was:

```text
Cannot read properties of undefined (reading 'toString')
```

That exact error happens when `JSON.parse(row.employees)` succeeds, but the parsed value does not contain `min` or `max`. Examples:

```text
"50"        -> parsed as number 50, parsed.min is undefined
"{}"        -> parsed.min is undefined
{"min":50}  -> parsed.max is undefined
```

A missing employees value would instead produce `Cannot read properties of null (reading 'min')`, and a normal range like `50-200` would produce a JSON syntax error. So the triggering CSV likely had an Employees column mapped with a value such as `50`, `{}`, or a partial structured range.

**Supporting evidence**

`employees` is stored as a plain optional string in Prisma: [schema.prisma](/home/dmytro/Github/phd-work/nextcrm-app-update/prisma/schema.prisma:1236)

The target update form also treats it as a plain string and even uses `50-200` as the placeholder: [UpdateTargetForm.tsx](/home/dmytro/Github/phd-work/nextcrm-app-update/app/[locale]/(routes)/campaigns/targets/components/UpdateTargetForm.tsx:347)

The CSV import preview only validates `last_name` or `company`; it does not validate or normalize `employees`, so the UI can allow the import and then fail in the server action: [ImportTargetsModal.tsx](/home/dmytro/Github/phd-work/nextcrm-app-update/components/modals/ImportTargetsModal.tsx:140)

Git history shows the failing parser was added in commit `9790f4c` titled `Handle legacy employee range imports` on May 1, 2026. The incident occurred on May 4, 2026. Before that commit, import stored `employees: row.employees || null`.

**Conclusion**

This is not a Next.js/App Router failure, database failure, or authentication issue. The import action regressed when legacy structured employee-range support was added without guarding non-legacy values. A CSV import with a mapped Employees column containing a JSON-valid but non-`{min,max}` value caused the action to throw and abort the request.

export const suggestMapping = async (headers: string[]) => {

        model: "gpt-4o-mini",
        messages: [
          {
            role: "system",
            content:
              "You are a data mapping assistant. Given a list of CSV column headers and a list of CRM field names, return a JSON object mapping each CSV header to the best matching CRM field, or null if no match. Only use the provided CRM field names as values. Return only valid JSON, no explanation.",
          },
          {
            role: "user",
            content: `CSV headers: ${JSON.stringify(headers)}\n\nCRM fields: ${JSON.stringify(TARGET_FIELDS)}\n\nReturn a JSON object like: { "CSV Header": "crm_field" or null }`,
          },
        ],
        response_format: { type: "json_object" },
        temperature: 0,
      });

      const content = completion.choices[0]?.message?.content;
      if (content) {
        const parsed = JSON.parse(content);
        const mapping: Record<string, string | null> = {};
        for (const header of headers) {
          const suggested = parsed[header];
          mapping[header] = suggested && TARGET_FIELDS.includes(suggested) ? suggested : null;
        }
        return { mapping };
      }
    } catch (err) {
      logger.error({ err: err }, "OpenAI mapping failed, falling back to fuzzy match");
    }
  }

  // Fallback: fuzzy match
  const mapping: Record<string, string | null> = {};
  for (const header of headers) {
    mapping[header] = fuzzyMatch(header);
  }
  return { mapping };

function normalizeCellValue(value?: string) {

function parseEmployeesValue(value?: string) {
  const parsed = JSON.parse(normalizeCellValue(value) || "null") as {
    min: number;
    max: number;
  };

  return `${parsed.min.toString()}-${parsed.max.toString()}`;

export async function importTargets(
  formData: FormData
): Promise<{ imported: number; skipped: number; errors: string[] }> {
  const session = await getSession();
  if (!session) throw new Error("Unauthorized");

  const file = formData.get("file") as File | null;
  if (!file) throw new Error("No file provided");

  const mappingRaw = formData.get("mapping") as string | null;
  const mapping: Record<string, string> = mappingRaw ? JSON.parse(mappingRaw) : {};

  const text = await file.text();
  const { data } = Papa.parse<Record<string, string>>(text, {
    header: true,
    skipEmptyLines: true,
  });

  const valid: any[] = [];
  const errors: string[] = [];

  data.forEach((rawRow, index) => {
    const row: Record<string, string> = {};
    if (Object.keys(mapping).length > 0) {
      Object.entries(mapping).forEach(([csvCol, targetField]) => {
        if (targetField && rawRow[csvCol] !== undefined) {
          row[targetField] = rawRow[csvCol];
        }
      });
    } else {
      Object.assign(row, rawRow);
    }

    const last_name = normalizeCellValue(row.last_name);
    const email = normalizeCellValue(row.email).toLowerCase();
    const mobile_phone = normalizeCellValue(row.mobile_phone);
    const company = normalizeCellValue(row.company);
    const company_website = normalizeCellValue(row.company_website);
    const companyDomain = company_website || company.toLowerCase() || null;
    const employees = parseEmployeesValue(row.employees);

    valid.push({
      last_name: last_name ?? "",

      email: email || null,
      mobile_phone: mobile_phone || null,
      office_phone: row.office_phone || null,
      company: row.company || null,

**Incident Report**

Most likely root cause: `actions/crm/targets/import-targets.ts` introduced an unsafe employee-range parser that assumes every imported `employees` value is JSON shaped as `{ "min": number, "max": number }`.

The crash occurs here: [import-targets.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/actions/crm/targets/import-targets.ts:11)

```ts
const parsed = JSON.parse(normalizeCellValue(value) || "null");
return `${parsed.min.toString()}-${parsed.max.toString()}`;
```

The server action calls this unconditionally for each CSV row at [import-targets.ts](/home/dmytro/Github/phd-work/nextcrm-app-update/actions/crm/targets/import-targets.ts:59), before row validation completes.

**Why this matches the incident**

The observed error was:

```text
Cannot read properties of undefined (reading 'toString')
```

That exact error happens when `JSON.parse(row.employees)` succeeds, but the parsed value does not contain `min` or `max`. Examples:

```text
"50"        -> parsed as number 50, parsed.min is undefined
"{}"        -> parsed.min is undefined
{"min":50}  -> parsed.max is undefined
```

A missing employees value would instead produce `Cannot read properties of null (reading 'min')`, and a normal range like `50-200` would produce a JSON syntax error. So the triggering CSV likely had an Employees column mapped with a value such as `50`, `{}`, or a partial structured range.

**Supporting evidence**

`employees` is stored as a plain optional string in Prisma: [schema.prisma](/home/dmytro/Github/phd-work/nextcrm-app-update/prisma/schema.prisma:1236)

The target update form also treats it as a plain string and even uses `50-200` as the placeholder: [UpdateTargetForm.tsx](/home/dmytro/Github/phd-work/nextcrm-app-update/app/[locale]/(routes)/campaigns/targets/components/UpdateTargetForm.tsx:347)

The CSV import preview only validates `last_name` or `company`; it does not validate or normalize `employees`, so the UI can allow the import and then fail in the server action: [ImportTargetsModal.tsx](/home/dmytro/Github/phd-work/nextcrm-app-update/components/modals/ImportTargetsModal.tsx:140)

Git history shows the failing parser was added in commit `9790f4c` titled `Handle legacy employee range imports` on May 1, 2026. The incident occurred on May 4, 2026. Before that commit, import stored `employees: row.employees || null`.

**Conclusion**

This is not a Next.js/App Router failure, database failure, or authentication issue. The import action regressed when legacy structured employee-range support was added without guarding non-legacy values. A CSV import with a mapped Employees column containing a JSON-valid but non-`{min,max}` value caused the action to throw and abort the request.
